# アプリケーション要件定義

このドキュメントは、HHKBカラーシミュレーターアプリケーションの主要な要件を定義します。

## 1. はじめに
HHKBキーボードのキーキャップを直感的にカスタマイズし、自分好みのカラースキームを視覚的に試行錯誤・保存できるWebアプリケーション。手軽にシミュレーションを行い、過去の作成データを管理することを目的とする。

## 2. 機能要件

### 2.1. トップページ（エディタ機能）
* **キーボード表示**:
    * 2Dレイアウト表示。
    * 初回はUS配列に対応し、将来的にJIS配列や他モデルへ拡張可能な設計とする。
* **個別カスタマイズ**:
    * 登録済みの6色のパレットから色を選択。
    * 各キーキャップを個別にクリック（タップ）して色を変更する機能。
* **保存機能**:
    * 現在のカスタマイズ状態に名前を付けて保存する機能。
    * 初回フェーズではブラウザのLocal Storage等を利用し、将来的にDB保存へ移行。

### 2.2. 一覧ページ（管理機能）
* **カスタマイズ一覧の閲覧**: 過去に保存したデザインをリストまたはカード形式で表示。
* **再編集**: 一覧から選択したデザインをトップページのエディタへ読み込む機能。
* **削除機能**: 不要になったカスタマイズデータを個別に削除（Delete）する機能。

## 3. 非機能要件
* **ユーザビリティ**:
    * 選択中のカラーが視覚的に強調され、直感的に操作できるUI。
    * 保存・削除時のフィードバック（完了メッセージ等）。
* **パフォーマンス**:
    * Inertia.jsを活用した、ページ遷移を感じさせないシームレスな操作感。
* **拡張性**:
    * レイアウト定義（座標情報）とユーザーの配色データを分離し、新しい配列（JIS等）の追加を容易にする。

## 4. 技術設計

### 4.1. レイアウト管理
* **Layout Definition (JSON)**:
    * 各キーを相対座標 (x, y) とサイズ (w) で定義。
    * 配列ごとの定義ファイルを切り替えることで、ロジックを変更せずに多種多様なキー配置を描画する。
* **保存データ構造**:
    * 「どのキーIDに、どの色を適用したか」の差分データのみを保持。
    * 形式例: `{ "layout_type": "US_HHKB", "colors": { "key_esc": "#FCDDE1", "key_q": "#FDFDFD" } }`

### 4.2. デフォルトカラーパレット
画像の解析に基づいた、HHKBの限定色に近いパステル調の6色を標準搭載する。
- 雪 `#FDFDFD`
- 蒲公英 `#FCEBB6`
- 桜 `#FCDDE1`
- 山葵 `#C1DEB7`
- 藤 `#D1C4E9`
- 青空 `#B3E5FC`

## 5. 技術要件
* **構成**: Laravel 13 + React + Inertia.js (Starter Kit)
* **言語**: TypeScript
* **スタイリング**: Tailwind CSS
* **データ保持**:
    * フロントエンド: React Context または State による状態管理。
    * ストレージ: LocalStorage (Phase 1) / PostgreSQL (Phase 2)。

## 6. 将来の検討事項
* **全モデル対応**: JIS配列対応、および筐体色（白・墨・雪）の切り替え機能。
* **ユーザー認証**: Laravel Breeze/Jetstreamによるログイン機能の実装。
* **クラウド同期**: ログインユーザーのデータをDBへ保存し、デバイス間で共有。
* **シェア機能**: カスタマイズ結果のURL発行、SNS共有機能。
* **3D表示**: Three.js等を用いた立体的なシミュレーション。
